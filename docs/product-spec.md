# 천원빵 – 제품 스펙 & DB 요약

## 1. 한 줄 요약

**KOSPI 지수를 00.00 단위로 예측해서, 15:30 종가 기준 1등한 사람에게 당일 참가비(빵)를 몰아주는 모바일 웹 게임. 실제 입출금은 수기(관리자), PG 없음.**

---

## 2. 제품 스펙 (세부)

| 항목 | 내용 |
|------|------|
| **단위** | 1빵 = 1,000원 (표시·정산만, 앱 내에서는 “빵” 단위로만 사용) |
| **실제 돈** | 입금·출금은 **수기**. PG/카카오페이 등 결제 API 없음. 관리자가 현금/계좌 입금 확인 후 앱에서 잔액만 반영. |
| **예측** | 회원이 **오전 9시 30분까지** 당일 KOSPI 지수를 **00.00** 단위로 입력. (업/다운 여부 + 수치 예측) |
| **순위** | **실시간(또는 일정 간격) 시세**와 예측값을 비교해서 **차이가 적은 순**으로 순위 집계. 장중에는 이 순위로 화면 표시. |
| **1등 결정** | **15:30 장 마감** 시점의 **종가**를 기준으로 최종 1등 확정. |
| **몰빵** | 당일 참가한 사람들이 낸 빵(참가비)이 “풀”로 모이고, 그 **전액을 1등에게 지급** (앱 잔액으로 적립). 1등은 나중에 수기로 출금 요청 가능. |
| **제작 형태** | **모바일 웹** (반응형 웹으로 모바일에서 사용). |

---

## 3. 흐름 정리

1. **참가**  
   - 회원이 당일 9:30 전에 **예측(00.00)** 입력.  
   - 예측 등록 시 **참가비 1빵** 차감 (잔액 부족 시 예측 불가 등 규칙은 앱에서 정의).

2. **장중**  
   - 실시간(또는 N분마다) KOSPI 시세 조회 → 예측값과 비교 → **순위 집계** → 모바일 웹에 순위/차이 표시.

3. **15:30 장 마감**  
   - 해당일 **종가**를 DB(또는 관리자 입력)에 반영 → **최종 1등** 계산 (종가와 예측값 차이 최소 1명).

4. **몰빵**  
   - 당일 참가비로 모인 빵(참가자 수 × 1빵)을 **1등 계좌에 일괄 입금**.  
   - 거래내역에는 “당일 게임 1등 몰빵”으로 기록.

5. **입금(충전)**  
   - 회원이 현금/계좌로 관리자에게 입금 → 관리자가 확인 후 앱에서 해당 회원 **잔액 수기 증가** (충전 처리).

6. **출금**  
   - 1등 등 잔액이 쌓인 회원이 “출금 요청” → 관리자가 수기로 계좌 이체 등 진행 후, 앱에서 해당 회원 **잔액 수기 감소** (출금 처리).

---

## 4. DB에 필요한 것 (테이블 단위)

PG/결제 API가 없고, “빵” 단위 + 수기 입출금 + 1등 몰빵이라서 아래 구성이면 충분합니다.

---

### 4.1 회원 (members)

- **역할**: 로그인, 참가자 구분, 계좌와 1:1 연결.
- **필요한 이유**: 누가 예측했는지, 누가 1등인지, 누 계좌에 충전/출금/몰빵을 넣을지 구분하려면 회원 테이블이 필요함.
- **핵심 컬럼**: id, name, email, password_hash, (관리자 여부 role 등 선택), created_at, updated_at.

---

### 4.2 종가관리 마스터 (closing_price_master)

- **역할**: **15:30 장 마감 시점의 KOSPI 종가**를 날짜별로 1건씩 저장.
- **필요한 이유**: 최종 1등을 “종가와 예측값 차이”로 정하려면, 그날의 “공식 종가”가 DB에 있어야 함. 실시간 시세는 API로만 조회해도 되고, 필요하면 스냅샷 테이블을 따로 둘 수 있음.
- **핵심 컬럼**: id, trade_date(거래일), kospi_close(종가, 00.00), registered_at, created_at, updated_at.

---

### 4.3 예측 마스터 (prediction_master)

- **역할**: 회원이 **어느 날(trade_date)**에 **몇으로 예측했는지(00.00)** 저장. 1인 1일 1예측.
- **필요한 이유**: 순위·1등 계산의 입력 데이터. “실시간과 비교”는 이 예측값 + 실시간 시세로 계산하면 됨.
- **핵심 컬럼**: id, member_id, trade_date, predicted_value(00.00), submitted_at, created_at, updated_at.  
  - **UNIQUE(member_id, trade_date)** 로 같은 날 중복 예측 방지.

---

### 4.4 계좌 마스터 (account_master)

- **역할**: 회원당 **빵 잔액 1개**. 단위는 “빵”(1빵=1000원이지만 앱에서는 빵만 저장).
- **필요한 이유**: 참가비 차감, 1등 몰빵 입금, 수기 충전/출금을 모두 “잔액 증감”으로 처리하려면 회원별 계좌 1개가 필요함.
- **핵심 컬럼**: id, member_id(UNIQUE), balance(빵 단위, 기본 0), created_at, updated_at.

---

### 4.5 계좌 거래내역 (account_transactions)

- **역할**: 잔액이 **왜, 언제, 얼마나** 바뀌었는지 전부 기록. (충전, 참가비 차감, 1등 몰빵, 수기 출금, 관리자 조정 등)
- **필요한 이유**: 감사 추적, “당일 풀 금액 = 1등에게 간 금액” 검증, 출금/충전 이력 확인. 잔액은 이 내역의 합으로도 재계산 가능하게 두는 게 안전함.
- **핵심 컬럼**: id, account_id, amount(양수=입금, 음수=출금), transaction_type, ref_trade_date(해당일 게임일 때), memo, created_at.
- **transaction_type 예시**  
  - `ENTRY_FEE` : 참가비(1빵) 차감 → 당일 풀에 포함.  
  - `WIN_POOL` : 당일 1등 몰빵 입금.  
  - `CHARGE_MANUAL` : 수기 충전(관리자가 입금 확인 후).  
  - `WITHDRAW_MANUAL` : 수기 출금(관리자가 이체 후 잔액 차감).  
  - `ADJUST` : 관리자 조정(예외 보정 등).

---

### 4.6 게임 라운드 (game_rounds) — 권장

- **역할**: **날짜별 게임 1건**. 그날 예측 마감 여부(9:30), 종가 반영 여부(15:30 이후), **당일 1등 member_id**, 당일 풀 금액(또는 참가 인원) 등을 캐시.
- **필요한 이유**: “오늘 1등이 누구인지”, “이미 몰빵 처리했는지”를 빠르게 조회하고, 실시간 순위와 최종 결과를 같은 “오늘 게임” 개념으로 묶기 좋음. 없어도 종가·예측 테이블만으로 1등·풀 계산은 가능하지만, 있으면 화면·배치 로직이 단순해짐.
- **핵심 컬럼**: id, game_date, status(open/closed/finalized), closing_price_master_id, winner_id, (pool_amount 또는 participant_count 등 선택), created_at, updated_at.

---

### 4.7 (선택) 빵 마스터 (bread_master)

- **역할**: 1빵 = 1,000원 같은 **단가·표시명** 관리. 앱에서 “1빵 = 1,000원” 문구, 정산 시 금액 계산에 사용.
- **필요한 이유**: 필수는 아니고, 상수로 두어도 됨. 나중에 단가 변경·다른 상품 추가할 가능성이 있으면 마스터 테이블로 두면 유리함.

---

## 5. 실시간 순위 & 15:30 1등

- **실시간 순위**  
  - DB: `prediction_master`(당일 trade_date) + `closing_price_master`(전일 종가 또는 당일 실시간은 API).  
  - 실시간 시세는 **외부 API**로 조회(한국거래소/증권 API 등). 그때그때 “현재 시세”와 각 예측값의 차이 `ABS(predicted_value - current_price)`로 정렬해서 순위 표시.  
  - 매 시각마다 DB에 실시간 시세를 넣을 필요는 없고, “현재 시세 한 개”만 API로 가져와서 순위만 계산해도 됨. 필요하면 `kospi_snapshots(timestamp, value)` 같은 테이블로 장중 스냅샷을 남길 수 있음.

- **15:30 1등**  
  - `closing_price_master`에 해당일 `kospi_close`가 들어온 뒤,  
  - `prediction_master`에서 같은 trade_date인 행들에 대해 `ABS(predicted_value - kospi_close)`가 **최소인 회원 1명**을 1등으로 확정.  
  - 동점이면 규칙 하나 정해서 처리(예: 먼저 제출한 사람 우선).

---

## 6. 몰빵 처리 (DB 관점)

- 당일 **참가비 차감**  
  - 예측 등록 시: `account_transactions`에 `transaction_type = ENTRY_FEE`, `amount = -1`, `ref_trade_date = 오늘` INSERT 후, 해당 회원 `account_master.balance -= 1`.
- **풀 금액**  
  - 당일 `ref_trade_date = 오늘` 이고 `transaction_type = ENTRY_FEE` 인 건들의 `amount` 절댓값 합 = 당일 풀(빵 개수). (1인 1빵이면 참가자 수와 같음.)
- **1등 몰빵**  
  - 15:30 이후 1등 확정되면: 1등 회원의 `account_master`에 대해 `account_transactions`에 `transaction_type = WIN_POOL`, `amount = + (당일 풀 금액)`, `ref_trade_date = 오늘` INSERT 후, `account_master.balance += (당일 풀 금액)`.
- **중복 지급 방지**  
  - `game_rounds`에 winner_id와 “몰빵 처리 완료” 플래그(또는 finalized 시점에 한 번만 WIN_POOL 넣는 로직)를 두면, 같은 날 두 번 몰빵이 들어가는 일을 막을 수 있음.

---

## 7. 수기 입출금 (DB 관점)

- **충전**  
  - 관리자: 해당 회원 계좌에 `transaction_type = CHARGE_MANUAL`, `amount = +N` INSERT, `account_master.balance += N`.
- **출금**  
  - 관리자가 실제 이체 후: 해당 회원 계좌에 `transaction_type = WITHDRAW_MANUAL`, `amount = -N` INSERT, `account_master.balance -= N`.
- PG/API 없이 위 두 가지 타입만 있으면 “실제 돈은 수기, 앱에는 빵 잔액만 반영”하는 구조가 됨.

---

## 8. 모바일 웹에서 필요한 것 (DB와 연결)

- **9:30 전**: 오늘 예측 등록 + 참가비 1빵 차감 가능 여부(잔액 체크).
- **장중**: 실시간(또는 주기) KOSPI 시세 API 조회 → 예측과 비교한 **순위** 표시.
- **15:30 이후**: 해당일 **종가** 표시, **최종 1등** 표시, (관리자) 몰빵 처리 버튼 또는 배치 한 번 실행.
- **공통**: 내 잔액, 내 예측 이력, 거래내역(충전/참가비/몰빵/출금), (관리자) 수기 충전/출금 처리 화면.

위 화면들이 필요한 데이터는 모두 위 5~6개 테이블(회원, 종가, 예측, 계좌, 거래내역, 게임라운드)로 커버 가능합니다.

---

이 스펙을 기준으로 `database-design.md`의 테이블 정의·거래 유형을 맞춰 두면, “PG 없이 빵 단위 + 수기 입출금 + 1등 몰빵” 구조를 DB에 그대로 반영할 수 있습니다.
